# Personal Life Blog PRD

状态：Draft v0.1

## 1. 项目目标

构建一个界面简洁、易用、适合长期记录生活的个人博客。网站以静态网页形式发布到 GitHub Pages，通过 Markdown 和图片文件管理内容。

这个项目同时作为学习 Git 和 GitHub 的实践项目。在日常维护博客的过程中，逐步熟悉 Git 基本命令、GitHub 仓库管理、issue、branch、Pull Request、GitHub Actions 和 GitHub Pages 发布流程。

## 2. 产品定位

这是一个以个人记录为核心的生活博客，而不是内容平台、作品集或社交产品。

第一版重点是做出稳定的网站形态和清晰的内容结构，让后续可以持续写文章、记录照片，并在未来接入更方便的内容更新方式，例如 Obsidian、手机快捷记录或 iMessages 灵感收集。

## 3. 第一版范围

第一版只包含三个主要版块：

1. 文章页，也是首页
2. 照片页
3. 个人说明页

第一版不做后台、不做网页上传、不做评论、不做站内搜索、不做访问统计。

## 4. 页面结构

### 4.1 文章页 / 首页

首页即文章目录，展示长文本记录的入口，不展示文章正文或正文摘要。

内容按发布时间倒序排列。每篇文章在目录中展示：

- 标题
- 日期
- 语言
- 标签，可选

点击文章后进入文章详情页。

文章详情页展示：

- 标题
- 发布日期
- 正文
- 标签，可选
- 语言切换入口，可选，仅当存在翻译版本时展示

### 4.2 照片页

照片页用于记录照片和简单描述，定位为轻量生活记录。

每条照片记录可以包含：

- 一张或多张照片
- 日期
- 简短描述
- 地点，可选
- 语言
- 语言切换入口，可选，仅当存在翻译版本时展示

第一版照片页优先采用简洁时间流或极简网格。重点是照片和文字本身，不做复杂相册系统。

### 4.3 个人说明页

个人说明页用于展示：

- 我是谁
- 为什么做这个博客
- 这个博客会记录什么
- 联系方式或 GitHub 链接，可选

文案保持简短、自然，不做营销式介绍。

## 5. 信息架构

中文为默认语言，英文使用 `/en/` 路径。

建议路径：

```text
/                 中文文章首页
/photos/          中文照片页
/about/           中文关于页

/en/              英文文章首页
/en/photos/       英文照片页
/en/about/        英文关于页
```

文章路径建议：

```text
/posts/example-slug/
/en/posts/example-slug/
```

照片记录路径可以第一版先不单独开放详情页，直接在照片页展示。后续如果照片记录变复杂，再增加独立详情页。

## 6. 内容类型

文章和照片分成两个内容类型。文章负责长文本，照片负责轻量记录。

### 6.1 文章 Post

文章用于长文本记录。

建议字段：

```text
title
date
lang
slug
draft
description
tags
translationKey
```

字段说明：

- `title`：标题
- `date`：发布日期
- `lang`：内容语言，例如 `zh` 或 `en`
- `slug`：URL 使用的稳定英文标识
- `draft`：是否草稿
- `description`：列表页和 SEO 使用的摘要
- `tags`：标签，可选
- `translationKey`：中英文版本的关联标识，可选

### 6.2 照片 Photo

照片记录用于图片和简短描述。

建议字段：

```text
title
date
lang
slug
draft
description
images
location
translationKey
```

字段说明：

- `title`：照片记录标题，可简短
- `date`：记录日期
- `lang`：内容语言，例如 `zh` 或 `en`
- `slug`：稳定英文标识
- `draft`：是否草稿
- `description`：照片描述
- `images`：图片路径列表
- `location`：地点，可选
- `translationKey`：中英文版本的关联标识，可选

## 7. 中英文策略

网站支持中文和英文，但不强制每条内容都有双语版本。

原则：

1. 中文为默认语言。
2. 英文内容放在 `/en/` 路径下。
3. 每篇文章或照片记录可以只有中文、只有英文，或同时存在中英文版本。
4. 如果存在翻译版本，通过 `translationKey` 关联。
5. 如果不存在翻译版本，不显示语言切换入口。

这样可以避免每次写内容都必须翻译，降低长期维护压力。

## 8. 视觉与排版

整体风格为极简文字型。

设计原则：

- 内容优先
- 留白充足
- 少装饰
- 少颜色
- 少动画
- 不做复杂卡片化布局
- 不做营销式首页

### 8.1 字体

第一版使用系统字体栈，不引入外部字体。

目标是在 macOS 和 iOS 上尽量接近 Apple 原生字体体验，在其他系统上使用合适的系统 fallback。

建议字体栈：

```css
font-family:
  -apple-system,
  BlinkMacSystemFont,
  "SF Pro Text",
  "PingFang SC",
  "Hiragino Sans GB",
  "Microsoft YaHei",
  "Noto Sans CJK SC",
  sans-serif;
```

### 8.2 排版建议

建议排版参数：

```text
正文大小：16px 或 17px
正文行高：1.75 - 1.9
文章内容宽度：640px - 720px
标题字重：600
正文颜色：接近 #1d1d1f
次要文字颜色：接近 #6e6e73
背景色：#ffffff 或 #fbfbfd
```

第一版不使用复杂主题系统。深色模式、主题切换可以作为后续功能。

## 9. 编辑与发布流程

第一版不做后台管理系统。内容通过 Markdown 和图片文件维护。

文章和照片使用相似的编辑流程：

```text
新建 Markdown 内容文件
填写标题、日期、语言、slug 等信息
写正文或照片描述
添加图片文件
本地预览
提交 commit
push 到 GitHub
GitHub Pages 自动发布
```

第一版可以先使用手动模板创建内容。后续可以增加命令辅助生成文章和照片记录。

未来可演进方向：

- Obsidian 写作同步
- 手机快捷记录
- iMessages 灵感收集
- 自动生成草稿
- 照片自动整理

这些能力不进入第一版，但第一版内容结构应保持清晰和稳定，方便后续接入。

## 10. 技术方向

推荐使用 Astro 构建。

选择 Astro 的原因：

- 适合静态网站
- 支持 Markdown 内容
- 适合极简博客
- 可扩展双语路由
- 可部署到 GitHub Pages
- 后续容易扩展照片页、归档、RSS、搜索等功能

发布方式：

```text
Astro build
GitHub Actions
GitHub Pages
```

模板参考：

- Astro 官方 Blog 模板
- AstroPaper
- Astro i18n Starter
- Accessible Astro Starter

第一版优先参考结构和最佳实践，不直接套用过重主题。

## 12. 暂不做功能

第一版明确不做：

- 后台管理系统
- 登录系统
- 评论系统
- 站内搜索
- RSS
- 访问统计
- 复杂主题切换
- 复杂动画
- 数据库
- 服务端接口
- CMS
- 网页上传图片
- 自动从 Obsidian 或 iMessages 导入

这些功能可以后续按真实需求逐步加入。

## 13. 第一版验收标准

第一版完成时应满足：

1. 网站可以本地运行和预览。
2. 网站可以部署到 GitHub Pages。
3. 首页展示文章列表。
4. 文章详情页可以正常打开。
5. 照片页可以展示照片和简短描述。
6. 关于页可以展示个人说明。
7. 中文路径和英文路径结构清晰。
8. 单语内容和双语内容都能被合理处理。
9. 字体和排版符合极简、清晰、类似 Apple 原生体验的方向。
10. 内容文件结构能支持后续 Obsidian、手机记录或自动化导入。

## 14. 后续待讨论

以下问题后续实现前可以继续确认：

1. 第一版是否需要文章标签页。
2. 照片页采用时间流还是网格。
3. 是否需要照片详情页。
4. 是否第一版就开启 RSS。
5. 是否需要自定义域名。
6. 首页每页展示多少篇文章。
7. 是否需要草稿内容本地可见、线上不可见。
